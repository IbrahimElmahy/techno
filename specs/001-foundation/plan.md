# Implementation Plan: Foundation (Shared Base)

**Branch**: `001-foundation` | **Date**: 2026-06-25 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-foundation/spec.md`

## Summary

Build the shared backbone every later domain posts to: server-side RBAC with branch/rep isolation
(deny-by-default), the organization tree (head office в†’ branches/governorates в†’ territories в†’
warehouses + per-rep custodies), customers with ledger-derived receivable accounts, and вЂ” at the
center вЂ” an **immutable double-entry ledger** from which all treasury, custody, and customer-account
balances are *derived, never stored standalone*. Every ledger entry supports first-class
reversal-linkage (a mirror entry), and all write/security actions are audited.

Technical approach: Python + FastAPI exposing an OpenAPI contract (the versioned shared contract per
Principle II), SQLAlchemy ORM over MySQL/MariaDB, Alembic migrations, pytest. The ledger, reversal
symmetry, balance derivation, and access-control denials are built **test-first** (Principle X).
Products, invoices, stock movements, loyalty, offline sync, and employees are explicitly out of
scope; the schema must not over-commit to their later design.

## Technical Context

**Language/Version**: Python 3.12
**Primary Dependencies**: FastAPI, SQLAlchemy 2.x, Alembic, Pydantic v2, passlib[bcrypt],
python-jose (JWT), uvicorn
**Storage**: MySQL 8 / MariaDB 10.6+ (InnoDB, utf8mb4, `DECIMAL(18,2)` for money)
**Testing**: pytest, pytest-asyncio, httpx (ASGI test client), testcontainers or a disposable MySQL
schema for integration tests
**Target Platform**: Linux server (containerized); API consumed by the Flutter app and back-office web
**Project Type**: Web service (backend-only in this feature; mobile/web clients are separate features)
**Performance Goals**: Interactive back-office latency p95 < 300 ms for CRUD; balance-derivation
queries p95 < 500 ms at foundation scale
**Constraints**: All money is EGP `DECIMAL(18,2)`; balances MUST be ledger-derived; posted ledger
rows immutable; all access decisions server-side; Arabic/RTL + EGP presentation
**Scale/Scope**: Single company вЂ” tens of branches, low-hundreds of users, low-thousands of
customers, ledger entries growing unbounded over time (design for append-only growth + indexing)

dedicated failing-first unit suites.

## Complexity Tracking

> These are deliberate decisions that add structure beyond the simplest possible approach. They are
> **justified**, not constitution violations вЂ” recorded here for transparency per the plan template.

| Decision | Why Needed | Simpler Alternative Rejected Because |
|----------|------------|--------------------------------------|
| Double-entry ledger with separate `LedgerEntry` (header) + `LedgerLine` (debit/credit legs) + `Account` rows, balances computed on read | Principle VI requires every balance to be ledger-derived and books to always balance; a header/lines split is the standard way to guarantee ОЈdebits = ОЈcredits per entry and to attach N legs (treasury, custody, customer) to one event | A single flat "transactions" table with a signed amount cannot enforce balanced double-entry, cannot represent multi-leg postings (e.g., customer credit purchase touching receivable + treasury/custody), and would tempt stored running balances that drift from truth |
| Permission resolved by a deny-by-default RBAC layer (roleв†’permission map + scope predicates) injected as a FastAPI dependency on every route | Principle VII demands server-side enforcement on every endpoint with branch/rep isolation; centralizing it prevents per-endpoint drift and makes "denied unless granted" the default | Ad-hoc `if role == ...` checks inside handlers are easy to forget on a new endpoint (silent over-exposure), can't be tested uniformly, and scatter the security model |

> Note: Materialized/cached balances are intentionally **out of scope** now. If read performance later
> demands it, a cache MUST be a derived projection of the ledger (rebuildable, never authoritative) and
> would get its own Complexity Tracking entry at that time.
